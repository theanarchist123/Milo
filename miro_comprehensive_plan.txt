DESIGN PHILOSOPHY
─────────────────
Miro's UI should feel like a premium academic command center — not a student
productivity app. Think Bloomberg Terminal meets Notion meets Linear. Every
screen should feel purposeful, fast, and alive. Dark-first design with a
warm amber/cream accent palette to signal intelligence and trust.

COLOR PALETTE
─────────────
Primary Background   : #0A0A0F (near-black, deep space)
Secondary Surface    : #111118 (card backgrounds)
Tertiary Surface     : #1A1A24 (elevated panels)
Primary Accent       : #F5C842 (Miro amber — used for highlights, CTAs)
Secondary Accent     : #6366F1 (indigo — secondary actions, AI states)
Success              : #10B981 (teal green)
Danger               : #EF4444 (red)
Text Primary         : #F1F0ED (off-white)
Text Secondary       : #8B8A8F (muted gray)
Text Tertiary        : #4A4A55 (dim gray)
Border               : rgba(255,255,255,0.06)

TYPOGRAPHY SCALE
────────────────
Display  : Inter 700 · 48–64px · letter-spacing -0.04em
Heading1 : Inter 600 · 32px · letter-spacing -0.02em
Heading2 : Inter 500 · 24px
Body     : Inter 400 · 15px · line-height 1.7
Caption  : Inter 400 · 12px · text-secondary color
Code     : JetBrains Mono 400 · 13px

ANIMATION PRINCIPLES (AnimateUI Library)
─────────────────────────────────────────
Every element must feel alive. Rules:
· Entrance: fade-up (translateY 20px → 0, opacity 0 → 1) · 400ms ease-out
· Stagger children: 60ms delay between each list item
· Hover: scale(1.02) on cards · 150ms
· Page transitions: crossfade + slight blur · 300ms
· Loading states: shimmer skeleton, never spinners
· Success states: confetti burst (AnimateUI) then settle
· AI processing: pulsing amber glow on active card
· Number counters: animate up on mount (AnimateUI Counter)
· Scroll-triggered: elements animate in as user scrolls down

SHADCN/UI COMPONENT USAGE GUIDELINES
──────────────────────────────────────
Use shadcn/ui as base, then layer AnimateUI motion on top:
· Dialog     → Use for file preview modals (blur backdrop, slide-up motion)
· Sheet      → Right-side panel for task details
· Card       → All content cards (override with dark surface colors)
· Badge      → Classification labels (Assignment / Notes / Announcement)
· Progress   → Processing pipeline progress bar
· Tabs       → Dashboard section switching
· Command    → Global search palette (CMD+K)
· Tooltip    → Hover info on all icons
· Skeleton   → Loading state for every data-fetched element
· Toast      → Notification system (top-right, animated slide-in)
· Avatar     → User profile photo (from Firebase Auth)
· Separator  → Section dividers (very low opacity)
· ScrollArea → All scrollable lists (custom scrollbar)

UNSPLASH API USAGE
──────────────────
Unsplash provides dynamic academic imagery throughout the app.
Usage points:
· Login/Onboarding Screen   : Full-screen background image · query="library study"
· Empty States              : Contextual images when no data yet
· Course Cards              : Subject-based images · query="{subject name}"
· Feature illustration banners inside onboarding walkthrough
· Profile background header : query="abstract minimal"
API endpoint : https://api.unsplash.com/photos/random?query={term}&orientation=landscape
Key rule     : Always show images with a gradient overlay (bottom black fade)
               so text on top is always legible.
Cache        : Store fetched Unsplash URLs in localStorage · refresh every 24h

· /                → Landing / Login (redirect to /dashboard if authed)
· /dashboard       → Command Center
· /emails          → Inbox View
· /classroom       → Classroom Hub
· /files           → The Vault
· /process/:taskId → Processor View
· /settings        → Settings
All routes wrapped in AuthGuard component that checks Firebase auth state.

STATE MANAGEMENT (Zustand):
· authStore   : user object, OAuth tokens, loading state
· taskStore   : active tasks, task history
· uiStore     : sidebar open/close, active filters, search query

FIREBASE INTEGRATION ON FRONTEND:
· Firebase SDK initialized in /lib/firebase.ts
· Google Sign-In with custom scopes added at auth time
· After sign-in: extract access_token from credential
· Store access_token in Zustand (not localStorage for security)
· Send access_token to backend as Authorization header on all requests
· Firestore: real-time listeners on /tasks/{userId} collection
· Storage: use getDownloadURL() to get signed links for generated files

UNSPLASH INTEGRATION:
· Custom hook: useUnsplash(query) → returns image URL
· Caches results in a Map to avoid duplicate requests
· Always fetches with orientation=landscape and w=1200 for quality
· Shows shimmer skeleton while image loads, then crossfade reveal

ANIMATEUI INTEGRATION:
· Install AnimateUI alongside Framer Motion (its dependency)
· Create /components/animated/ wrappers:
    – AnimatedCard  : fade-up entrance on mount
    – AnimatedList  : staggered children entrance
    – AnimatedBadge : pop-in on mount
    – AnimatedCounter : number count-up animation
    – AnimatedProgress : smooth progress bar with glow
· All page-level components wrapped in AnimatedPage (crossfade transition)
· Use AnimateUI's useInView hook for scroll-triggered animations

API COMMUNICATION:
· All backend calls through a central apiClient (axios instance)
· Base URL set from environment variable VITE_BACKEND_URL
· Auth header injected via axios interceptor from Zustand authStore
· Error handling: 401 → re-prompt Firebase sign-in, 500 → toast error

  PART 4 — BACKEND ARCHITECTURE PLAN

BACKEND PHILOSOPHY
──────────────────
The backend is a headless AI pipeline engine. It receives tasks from the
frontend, executes multi-step agent workflows, writes progress to Firestore
in real time, and stores results in Firebase Storage. It never "holds"
sensitive email content longer than the current task needs.

PROJECT STRUCTURE
─────────────────
/backend
  /api
    /routes      → FastAPI route files per domain
    /middleware  → Firebase token verification, CORS, rate limiting
  /agents
    /orchestrator → LangGraph graph definition and state machine
    /nodes        → individual agent node functions (one per pipeline step)
  /services
    /gmail        → Gmail API client — fetch, read, download attachments
    /classroom    → Classroom API client — courses, coursework, materials
    /drive        → Drive API client — file download by fileId
    /extractor    → document text extraction logic per file type
    /ocr          → OCR.space API client wrapper
    /gemini       → Gemini API client — classification + generation prompts
    /rag          → ChromaDB setup, chunk storage, retrieval logic
    /writer       → DOCX and PDF output generation
    /storage      → Firebase Storage upload and signed URL generation
    /notifier     → Gmail API send — notification email
  /workers
    /celery_app   → Celery + Redis configuration
    /tasks        → background task definitions (one per major workflow)
  /models         → Pydantic data models for all requests and DB documents
  /config         → environment variable loading, API key management
  main.py         → FastAPI app init, route registration, startup events

================================================================================
  PART 5 — BACKEND FEATURES (WORKFLOW & INSTRUCTIONS)
================================================================================

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
BACKEND FEATURE 01 — THE GATEKEEPER
Firebase Authentication Middleware
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CONCEPT: "Every request must prove who it is before anything happens"

PURPOSE:
Verify every incoming API request is from a legitimate authenticated user.
Extract the OAuth2 access token so downstream services can call Google APIs
on behalf of the user.

WORKFLOW:
1. Frontend sends request with header: Authorization: Bearer {firebase_id_token}
2. Backend middleware intercepts every request before it hits any route
3. Firebase Admin SDK verifies the ID token — rejects if expired or invalid
4. Extracts uid and email from decoded token
5. Looks up the user's stored OAuth2 access_token from Firestore
   (stored there at first sign-in by a dedicated /auth/store-token endpoint)
6. Attaches { uid, email, oauth_access_token } to request state
7. All downstream route handlers read from request.state — never from client

SECURITY RULES:
· ID token expires every 1 hour — Firebase handles silent refresh on frontend
· OAuth2 access token also expires — check expiry before using, refresh if needed
· Never log or print access_token in any log output
· Rate limit: max 100 requests per user per minute (FastAPI middleware)

FIRESTORE SCHEMA for user auth data:
Collection: /users/{uid}
Document fields:
  · email                 : string
  · display_name          : string
  · photo_url             : string
  · oauth_access_token    : string (encrypted at rest)
  · oauth_refresh_token   : string (encrypted at rest)
  · token_expiry          : timestamp
  · created_at            : timestamp
  · last_sync             : timestamp

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
BACKEND FEATURE 02 — THE HARVESTER
Gmail + Classroom Data Fetch Service
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CONCEPT: "Sweep every platform clean — leave nothing behind"

GMAIL FETCH WORKFLOW:
1. Use user's OAuth2 access token to initialize Gmail API client
2. Call users.messages.list with query filter:
   · Filter for academic keywords: assignment, homework, submission, deadline,
     notes, lecture, exam, test, project, due, class, course, grade
   · Limit to past 30 days (configurable)
   · Max 50 emails per sync (configurable)
3. For each message ID returned, call users.messages.get to get full message
4. Parse message payload:
   · Extract subject, sender, date, body (handle both text/plain and text/html)
   · Detect parts where mimeType is NOT text/* — these are attachments
5. For each attachment detected:
   · Record: filename, mimeType, attachmentId, size
   · If size < 10MB: immediately download via users.messages.attachments.get
   · Attachment data comes as base64 — decode to raw bytes
   · Save bytes temporarily in /tmp/ for processing
6. Store each email record in Firestore:
   Collection: /users/{uid}/emails/{message_id}
   Fields: subject, sender, date, body_text, attachment_list, status="fetched"

CLASSROOM FETCH WORKFLOW:
1. Initialize Classroom API client with user's OAuth2 token
2. Call courses.list → get all active courses for the student
3. For each course:
   a. Call courses.courseWork.list → get all assignments and tasks
   b. Call courses.announcements.list → get all teacher announcements
   c. Call courses.courseWorkMaterials.list → get all study materials
4. For each coursework/material item that has Drive file attachments:
   · Extract driveFile.id from the materials list
   · Call Drive API files.get(fileId, alt=media) to download the file
   · Save raw bytes to /tmp/ for processing
5. Store in Firestore:
   Collection: /users/{uid}/classroom/{courseId}/items/{itemId}
   Fields: title, type, due_date, attachments, description, status="fetched"

SYNC TRIGGER MODES:
· Manual  : User clicks "Sync Now" button in dashboard
· Webhook : Gmail Push Notification via PubSub (advanced, Phase 3)
· Polling : Celery periodic task runs every hour (simpler, Phase 1)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
BACKEND FEATURE 03 — THE DETECTIVE
Content Classification Service (Gemini)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CONCEPT: "Read between the lines — know what matters"

PURPOSE:
Before processing any email or attachment, Miro must understand what it is.
The Classifier reads the content and assigns a label that determines the
entire downstream pipeline path.

INPUT TO CLASSIFIER:
· Email subject line
· Email body (first 2000 chars)
· List of attachment filenames and types
· Source: Gmail or Classroom
· Course name (if from Classroom)

GEMINI PROMPT STRATEGY:
Send structured prompt to Gemini 2.0 Flash asking it to return pure JSON.
The JSON schema to request:
  {
    "type"         : "ASSIGNMENT" | "NOTES" | "ANNOUNCEMENT" | "EXPERIMENT" | "UNCLASSIFIED",
    "subject_area" : "Mathematics" | "Physics" | "Computer Science" | ... | "Unknown",
    "priority"     : "HIGH" | "MEDIUM" | "LOW",
    "has_deadline" : true | false,
    "deadline_text": "string or null — extracted deadline if found in text",
    "action_required": true | false,
    "summary_one_line": "one sentence describing what this item is"
  }

CLASSIFICATION RULES:
· ASSIGNMENT   : has submission deadline, requires student response/output
· NOTES        : lecture slides, reading material, study guides, reference docs
· ANNOUNCEMENT : informational, no action required, from teacher
· EXPERIMENT   : lab report template, experiment instructions, practical task
· UNCLASSIFIED : cannot confidently assign any category

ROUTING AFTER CLASSIFICATION:
· ASSIGNMENT   → Extractor → RAG → Assignment Generator → Writer → Notifier
· NOTES        → Extractor → RAG → Summarizer + Q&A Generator → Writer → Notifier
· ANNOUNCEMENT → Store in Firestore only → no file output generated
· EXPERIMENT   → Extractor → RAG → Experiment Report Generator → Writer → Notifier
· UNCLASSIFIED → Flag for manual review in dashboard

UPDATE FIRESTORE:
Update the email/item document with all classification fields.
Update status field from "fetched" to "classified".

BACKEND FEATURE 04 — THE DISSECTOR
Document Text Extraction Service
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CONCEPT: "Tear open every file — read what's inside"

PURPOSE:
Convert raw file bytes (PDF/DOCX/PPTX/images) into clean structured text
that the AI can process. Use the fastest and most accurate method per file type.
Fall back gracefully to OCR.space for scanned/image content.

PDF EXTRACTION WORKFLOW:
1. Open file with PyMuPDF
2. Extract text page by page
3. Check text quality: if total chars across all pages < 100:
   file is likely scanned or image-only → go to OCR fallback
4. If text is good: clean it (remove header/footer artifacts, extra whitespace)
5. Return: { pages: [{page_num, text, word_count}], total_text, page_count }

OCR.space FALLBACK WORKFLOW (for scanned PDFs or image files):
1. Triggered when PyMuPDF yields insufficient text
2. Convert PDF pages to JPEG images at 200 DPI using PyMuPDF's rendering
3. Send image bytes to OCR.space API:
   · Endpoint: https://api.ocr.space/parse/image
   · Parameters: language=eng, isTable=false, scale=true, OCREngine=2
   · Send as multipart form — base64 or direct file upload
4. Parse response: extract ParsedText from each page result
5. Concatenate all page results into full document text
6. Return same format as direct extraction above

DOCX EXTRACTION WORKFLOW:
1. Open with python-docx
2. Extract all paragraphs in document order
3. Detect and extract tables: rows and cells as structured data
4. Extract heading levels separately (useful for summarization structure)
5. Detect any embedded images (note their presence but skip content)
6. Return: { paragraphs, headings, tables, total_text }

PPTX EXTRACTION WORKFLOW:
1. Open with python-pptx
2. Iterate all slides in order
3. Per slide: extract all text frames and text boxes
4. Extract speaker notes for each slide (often contains the lecture explanation)
5. Detect slide layout (title slide, content slide, blank)
6. Return: { slides: [{slide_num, title, body_text, notes}], total_text }

POST-EXTRACTION:
· Store extracted text in Firestore document (compressed if large)
· Chunk text into segments of ~500 tokens each for RAG
· Embed each chunk using sentence-transformers (all-MiniLM-L6-v2)
· Store embeddings + chunks in ChromaDB under the user's namespace
· Update document status: "extracted"

TEXT QUALITY CHECKS:
· Minimum 100 chars to be considered successfully extracted
· Remove repeated whitespace, control characters, null bytes
· Detect language (if non-English, flag it — OCR.space supports 27 languages)

BACKEND FEATURE 05 — THE FORGE
AI Content Generation Engine (Gemini + RAG)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CONCEPT: "Transform raw knowledge into crafted academic output"

PURPOSE:
The heart of Miro. Takes extracted text + classification type and produces
a structured, high-quality academic document using Gemini 2.0 Flash + RAG.

RAG RETRIEVAL STEP (before every generation call):
1. Take the email subject or document title as the query
2. Convert query to embedding using sentence-transformers
3. Query ChromaDB for top 5 most semantically similar chunks
4. Append retrieved chunks to the generation prompt as context
5. This ensures Gemini generates content grounded in the actual document

GENERATION PATH A — ASSIGNMENT GENERATOR:
Triggered when: type = ASSIGNMENT or EXPERIMENT

Input prompt components:
· System instruction: you are an academic writing assistant for a student
· Assignment instructions (full extracted text)
· Retrieved context chunks from ChromaDB
· Format requirements: generate a complete structured assignment document

Gemini output structure to request:
  Title, Introduction, Objectives, Methodology/Approach,
  Main Body (multiple sections with headings), Conclusion,
  References placeholder, Word count estimate

Additional processing:
· If deadline detected: add due date to document header
· If subject area detected: use appropriate academic tone
· Generate a separate one-page "key concepts" summary for the student

GENERATION PATH B — NOTES PROCESSOR:
Triggered when: type = NOTES

Three separate Gemini calls (can run sequentially or parallel):

Call 1 — Summarizer:
· Input: full extracted text (up to 10,000 chars), top RAG chunks
· Output: structured summary with sections matching original document structure
· Length target: 20% of original length
· Format: heading → paragraph → key points list per section

Call 2 — Q&A Generator:
· Input: full extracted text
· Output: 10–15 question-answer pairs
· Mix of: factual recall, conceptual understanding, application questions
· Format: Q: [question] / A: [detailed answer]

Call 3 — Concept Explainer:
· Input: key terms detected in extracted text
· Output: simple plain-language explanation of each key concept
· Format: term → 2–3 sentence explanation a student can understand

All three outputs combined into one output document.

GENERATION PATH C — EXPERIMENT REPORT GENERATOR:
Triggered when: type = EXPERIMENT

Gemini generates a structured lab report template filled with content:
  Title, Aim, Hypothesis, Materials/Equipment, Procedure,
  Observations (blank table structure for student to fill),
  Results Analysis guidance, Conclusion template,
  Safety Precautions if relevant

GEMINI API CALL PARAMETERS:
· Model      : gemini-2.0-flash (latest)
· Temperature: 0.4 (balanced — creative but factual)
· Max tokens : 4000 per call
· Safety settings: block only HARM_BLOCK_THRESHOLD_UNSPECIFIED
· Response format: instruct plain text with markdown formatting

RATE LIMIT HANDLING:
· Gemini free: 15 RPM (requests per minute)
· If rate limit hit: exponential backoff — wait 4s, 8s, 16s then fail
· Log rate limit events to Firestore for monitoring
· Queue multiple tasks through Celery to avoid simultaneous calls

BACKEND FEATURE 06 — THE SCRIBE
Document Output Writer
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CONCEPT: "Give the student something they can actually hand in"

PURPOSE:
Convert AI-generated text content into professionally formatted DOCX and PDF
files ready for submission or study use.

DOCX GENERATION WORKFLOW:
1. Create new python-docx Document object
2. Apply custom document styles:
   · Normal: Calibri 12pt, 1.15 line spacing
   · Heading 1: Calibri Light 18pt, dark blue color, spacing before/after
   · Heading 2: Calibri 14pt, medium gray, spacing above
   · Title: Calibri 24pt bold, centered
   · Body text matches standard academic paper format
3. Insert document content:
   · Cover section: document title, student name placeholder, date, course name
   · Parse AI output (markdown-like structure) into proper Word paragraphs
   · Detect "##" headings → apply Heading styles
   · Detect "**text**" → apply bold
   · Detect bullet lists → Word list style
   · Detect table markers → create Word table with borders
4. Insert footer: page numbers (right-aligned) + "Generated by Miro" (left)
5. Insert header: course name + document type
6. Save as .docx file to /tmp/

PDF GENERATION WORKFLOW:
1. Build Jinja2 HTML template for the document type (assignment/notes/experiment)
2. Inject AI-generated content into template variables
3. Apply CSS styling:
   · Professional A4 layout
   · Academic font stack (Georgia for body, Helvetica for headings)
   · Proper margins (2.5cm all sides)
   · Subtle header/footer
   · Color accents matching document type
4. Render HTML to PDF using WeasyPrint
5. Save as .pdf file to /tmp/

FIREBASE STORAGE UPLOAD:
1. Upload both .docx and .pdf to Firebase Storage
   · Path: /outputs/{uid}/{task_id}/filename.docx
   · Path: /outputs/{uid}/{task_id}/filename.pdf
2. Get signed download URL for each file (valid for 7 days)
3. Store URLs in Firestore task document
4. Clean up /tmp/ files after upload succeeds

BACKEND FEATURE 08 — THE CONDUCTOR
LangGraph Agent Orchestrator
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CONCEPT: "The brain that connects all the parts in the right order"

PURPOSE:
LangGraph defines the entire processing pipeline as a state machine graph.
Each node in the graph is one service function. Edges have conditions that
determine which node runs next based on the current state.

GRAPH STATE OBJECT (passed between all nodes):
{
  task_id, uid, source_type (gmail/classroom),
  raw_content, attachment_path, attachment_type,
  extracted_text, classification, rag_chunks,
  generated_content, output_paths, notification_sent,
  current_step, error, retry_count
}

GRAPH NODES:
· fetch_data         → calls Gmail/Classroom/Drive services
· classify_content   → calls Gemini classifier
· extract_text       → calls extractor service (PyMuPDF or OCR.space)
· store_chunks       → stores text chunks in ChromaDB
· generate_content   → calls Gemini generator (routes by classification)
· write_files        → calls writer service (DOCX + PDF)
· upload_to_storage  → calls Firebase Storage uploader
· send_notification  → calls Herald (Gmail send)
· update_status      → writes final status to Firestore
· handle_error       → logs error, updates Firestore, optionally retries

GRAPH EDGES (conditional routing):
· After classify: if ANNOUNCEMENT → skip to update_status (no file needed)
· After extract: if text quality too low → handle_error (ask user to retry)
· After generate: if content empty → handle_error
· After any node: if error raised → handle_error

TASK LIFECYCLE IN FIRESTORE:
Status field updates as pipeline progresses:
  QUEUED → FETCHING → CLASSIFYING → EXTRACTING → GENERATING → WRITING → UPLOADING → DONE
  (or ERROR at any step)

Frontend listens to this status field via Firestore onSnapshot → updates
the pipeline visualizer in real time without polling.

CELERY INTEGRATION:
· LangGraph agent invocation wrapped in a Celery task
· Celery worker picks up task from Redis queue
· Multiple workers can process multiple tasks concurrently
· Task timeout set to 10 minutes per task
· Failed tasks retried up to 2 times with 30-second delay
